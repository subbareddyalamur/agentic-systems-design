OpenAI Assistants & Tool Use

📖 15 min read 📚 Topic 1 of 13

OpenAI's Assistants API provides a managed runtime for building AI agents. Rather than managing conversation state yourself, the API handles threads, messages, and tool execution through a structured lifecycle.

Architecture: Threads, Runs & Messages

The Assistants API is built around three core primitives:

  • Assistant — A configured agent with instructions, model selection, and tools
  • Thread — A conversation session that persists messages and context
  • Run — An execution of an assistant against a thread, producing responses
sequenceDiagram
    participant User
    participant Thread
    participant Run
    participant Assistant
    participant Tool

    User->>Thread: Add message
    User->>Run: Create run
    Run->>Assistant: Execute with thread context
    Assistant->>Tool: Call function
    Tool-->>Assistant: Return result
    Assistant-->>Run: Generate response
    Run-->>Thread: Append assistant message

Function Calling / Tool Use

OpenAI's function calling lets you define tools as JSON Schema, and the model decides when to invoke them. When a run enters the requires_action state, your code executes the function and submits results back.

Key Pattern: The model doesn't execute tools itself — it generates structured JSON arguments, your code runs the actual function, and you feed results back to continue the run.

Built-in Tools

Code Interpreter — Executes Python in a sandboxed environment. Great for data analysis, chart generation, and math. Files can be uploaded and processed.

File Search — Indexes uploaded documents and performs semantic retrieval (built-in RAG). Supports PDF, DOCX, TXT, and more.

Structured Outputs

Use response_format with a JSON Schema to guarantee the model returns valid, structured data. This eliminates parsing errors and makes tool outputs predictable.

API Key Management

Securely handling API keys is critical for production agent systems. These patterns apply to all LLM providers (OpenAI, Anthropic, AWS, Google):

  • Environment variables: Store keys in .env files (never commit them). Load with os.environ["OPENAI_API_KEY"] or python-dotenv
  • Secrets managers: In production, use AWS Secrets Manager, GCP Secret Manager, or Azure Key Vault for rotation and access control
  • Key rotation: Rotate keys regularly. Design your system so key changes don't require code deployments
  • Scoped keys: Use project-scoped or service-scoped keys with minimum required permissions
  • Rate limit awareness: Different API tiers have different rate limits. Implement client-side throttling to avoid 429 errors
Never hardcode API keys. Check your .gitignore includes .env, *.key, and any config files containing secrets. Use tools like git-secrets to prevent accidental commits.

Claude Tool Use & Computer Use

📖 15 min read 📚 Topic 2 of 13

Anthropic's Claude models support tool use through the Messages API. Unlike OpenAI's managed Assistants runtime, Claude's approach is stateless — you pass the full conversation and tool definitions each time, giving you more control over the execution loop.

Claude Messages API & Tool Use

You define tools with JSON Schema input definitions. When Claude decides to use a tool, it returns a tool_use content block. You execute the tool and pass results back as a tool_result message.

    Claude-->>App: tool_use content block
    App->>Tool: Execute function
    Tool-->>App: Return result
    App->>Claude: messages.create(tool_result)
    Claude-->>App: Final text response

Computer Use

Claude's Computer Use capability lets the model control a desktop environment — taking screenshots, moving the mouse, clicking, and typing. This enables automating GUI-based workflows that have no API.

Important: Computer Use runs in a sandboxed virtual environment. Never give it access to sensitive credentials or production systems without proper guardrails.

Model Context Protocol (MCP)

MCP is an open protocol that standardizes how AI models connect to external tools and data sources. Instead of each model having proprietary tool formats, MCP provides a universal interface — like USB for AI tools.

Claude Code

Claude Code is an agentic coding tool that runs in your terminal. It can read files, write code, run commands, and manage git — all through natural language. It uses tool use under the hood to interact with your development environment.

AWS Bedrock Agents

📖 12 min read 📚 Topic 3 of 13

Amazon Bedrock Agents is a fully managed service for building AI agents on AWS. It handles the orchestration loop, integrates with AWS services, and provides built-in guardrails for enterprise deployments.

Architecture

A Bedrock Agent combines a foundation model with action groups (tools), knowledge bases (RAG), and guardrails. The agent runtime manages the ReAct-style reasoning loop automatically.

Knowledge Bases

Bedrock Knowledge Bases provide managed RAG. You point to an S3 bucket with documents, choose an embedding model, and Bedrock handles chunking, embedding, indexing, and retrieval automatically.

Action Groups

Action groups are the "tools" for Bedrock agents. Each action group maps to a Lambda function with an OpenAPI schema defining the available operations. The agent decides which actions to invoke based on user queries.

Guardrails

Bedrock Guardrails filter inputs and outputs for: denied topics, content filters (hate, violence, etc.), PII detection and redaction, word filters, and contextual grounding checks.

Enterprise Advantage: Bedrock Agents integrate natively with IAM, CloudWatch, CloudTrail, and VPC — making compliance and auditing straightforward for enterprise teams.

Multi-Agent Frameworks

📖 18 min read 📚 Topic 5 of 13

Multi-agent frameworks enable you to compose multiple specialized AI agents into collaborative systems. Each framework takes a different approach to agent definition, communication, and orchestration.

CrewAI: Role-Based Collaboration

CrewAI models agents as crew members with roles, goals, and backstories. Tasks are assigned to agents, and a process (sequential or hierarchical) coordinates execution.

AutoGen: Conversation-Driven Agents

Microsoft's AutoGen models multi-agent systems as conversations. Agents communicate via messages, and group chats enable complex multi-party interactions with configurable speaker selection.

LangGraph: State Machine Agents

LangGraph models agent workflows as directed graphs with typed state. Nodes are functions, edges are transitions, and the state machine provides precise control over flow, including human-in-the-loop.

OpenAI Swarm: Lightweight Handoffs

Swarm is OpenAI's lightweight framework for agent handoffs. Agents can transfer control to other agents, enabling natural routing patterns. It's designed for simplicity — minimal abstraction over the base API.

Choosing a Framework: CrewAI for role-based teams, AutoGen for conversation-heavy workflows, LangGraph for precise state control, Swarm for simple handoff patterns.

Model Context Protocol (MCP)

📖 14 min read 📚 Topic 6 of 13

The Model Context Protocol (MCP) is an open standard for connecting AI models to external tools and data sources. It solves the N×M integration problem — instead of every model needing custom integrations with every tool, both sides implement one protocol.

Architecture

MCP has three components:

  • Host — The AI application (Claude Desktop, IDE, custom app)
  • Client — Maintains a 1:1 connection with a server (managed by the host)
  • Server — Exposes tools, resources, and prompts to the AI model
graph LR
    H[Host Application] --> C1[MCP Client 1]
    H --> C2[MCP Client 2]
    C1 --> S1[MCP Server:
File System]
    C2 --> S2[MCP Server:
Database]
    S1 --> FS[Local Files]
    S2 --> DB[PostgreSQL]

Transport Mechanisms

stdio — Server runs as a subprocess, communicating via stdin/stdout. Best for local tools.

Streamable HTTP — Server runs as an HTTP service. Supports remote servers and server-sent events for streaming.

Server Capabilities

MCP servers can expose three types of capabilities:

  • Tools — Functions the model can call (like database queries, API calls)
  • Resources — Data the model can read (like files, database records)
  • Prompts — Reusable prompt templates the user can invoke
Ecosystem: There are thousands of MCP servers available for databases, APIs, file systems, cloud services, and more. Check the MCP server registry for pre-built integrations.

Production Prompt Management

📖 12 min read 📚 Topic 7 of 13

In production AI systems, prompts are living artifacts that need versioning, testing, and optimization — just like code. Poor prompt management leads to regressions, inconsistency, and runaway costs.

Prompt Versioning & Registries

Store prompts as versioned templates in a central registry. Each change gets a version number, and you can roll back instantly if a new version degrades quality.

A/B Testing Prompts

Route a percentage of traffic to different prompt versions and measure quality metrics (accuracy, latency, user satisfaction). This lets you make data-driven decisions about prompt changes.

Prompt Injection Defenses

Prompt injection is when user input manipulates the system prompt. Defenses include: input sanitization, system prompt isolation, output validation, and the "sandwich" technique (repeating instructions after user input).

Security: Never trust user input in prompts. Always sanitize, validate, and monitor for injection attempts. Use structured outputs to constrain model responses.

Cost Optimization

Prompt costs scale with token count. Strategies: use shorter system prompts, cache common prefixes, choose cheaper models for simple tasks, and batch similar requests.

E2E Testing & CI/CD for AI

📖 12 min read 📚 Topic 8 of 13

Testing AI systems is fundamentally different from testing traditional software. LLM outputs are non-deterministic, quality is subjective, and regressions can be subtle. A robust testing strategy combines deterministic checks, statistical evaluation, and LLM-as-judge patterns.

Testing Pyramid for AI

LLM-as-Judge Evaluation

Use a strong model to evaluate outputs from your system. Define rubrics, score on multiple dimensions (accuracy, helpfulness, safety), and aggregate scores across a test dataset.

CI/CD Pipeline for AI

A typical pipeline: lint → unit tests → integration tests → eval suite → staging deploy → canary release → production. Include eval gates that block deployment if quality drops below thresholds.

Key Principle: Your eval suite is your most important asset. Invest more in building comprehensive evals than in any single model improvement.

Ethics, Bias & Responsible AI

📖 18 min read 📚 Topic 9 of 13

AI agents make decisions that affect real people. Understanding sources of bias, implementing fairness checks, and building responsible AI systems is not optional — it's a core engineering responsibility.

Sources of Bias

  • Training data bias — Historical data reflects past inequalities
  • Selection bias — Non-representative data collection
  • Measurement bias — Flawed proxies for what you're measuring
  • Aggregation bias — One model for diverse populations
  • Deployment bias — System used differently than intended

Fairness Metrics

Demographic parity: Equal positive prediction rates across groups. Equal opportunity: Equal true positive rates. Predictive parity: Equal precision across groups.

Important: No single fairness metric works for all situations. Different metrics can conflict — choosing the right one depends on the specific use case and its consequences.

Red Teaming & Safety

Red teaming means systematically trying to make your AI system fail or produce harmful outputs. This includes testing for: jailbreaks, harmful content generation, PII leaks, bias in outputs, and unintended tool use.

Responsible AI Frameworks

Key principles: transparency, fairness, safety, privacy, accountability, and human oversight. Implement these through: model cards, impact assessments, bias audits, and incident response plans.

Real-World AI Failures

Learning from past failures is essential for building safer systems:

  • Microsoft Tay (2016): A chatbot released on Twitter that learned from user interactions. Within 24 hours, coordinated users taught it to produce toxic content. Lesson: Unfiltered learning from adversarial users is dangerous. Input filtering and guardrails are mandatory.
  • Amazon Resume Screener (2018): An ML model trained on historical hiring data penalized resumes containing the word "women's" because past hires were predominantly male. Lesson: Historical data encodes historical biases. Audit features and outcomes for protected attributes.
  • Healthcare Algorithm Bias (2019): A widely-used algorithm for allocating healthcare resources used cost as a proxy for health need. Because Black patients historically had less access to healthcare (lower costs), the algorithm systematically under-served them. Lesson: Proxy variables can introduce hidden bias. Validate what your model actually measures.
  • ChatGPT Hallucination in Legal Filing (2023): A lawyer used ChatGPT to research case law. The model fabricated nonexistent cases with realistic citations. The lawyer submitted them to court. Lesson: LLM outputs must be verified. Never use AI-generated content as authoritative without human review.

Alignment & Guardrails Design

Alignment means ensuring AI systems behave according to human intentions and values. For agentic systems, this is especially critical because agents take actions with real-world consequences.

graph TD
    U[User Intent] --> G[Guardrails Layer]
    G --> A[Agent]
    A --> T[Tool Call]
    T --> V[Output Validation]
    V --> |Pass| R[Response to User]
    V --> |Fail| F[Fallback / Block]
    G --> |Blocked Input| B[Rejection Message]

Key guardrails patterns for production agents:

  • Input guardrails: Filter prompt injections, toxic content, and out-of-scope requests before they reach the model
  • Output guardrails: Validate model responses for accuracy, safety, and schema compliance before returning to users
  • Tool-use guardrails: Restrict which tools an agent can call, with what parameters, and require confirmation for destructive actions
  • Rate limiting: Prevent runaway agent loops that could consume excessive resources or take repeated harmful actions
  • Human-in-the-loop: Require human approval for high-stakes decisions (financial transactions, account deletions, etc.)
Defense in Depth: No single guardrail is sufficient. Layer input validation, output filtering, tool restrictions, and human oversight for robust safety.

Capstone: RAG-Powered Q&A System

📖 15 min read 📚 Topic 10 of 13

In this capstone, you'll build a complete RAG (Retrieval-Augmented Generation) system from scratch. The pipeline: ingest documents → chunk → embed → store → retrieve → generate answers with citations.

Architecture

graph LR
    D[Documents] --> C[Chunker]
    C --> E[Embedder]
    E --> VS[Vector Store]
    Q[User Query] --> QE[Query Embedder]
    QE --> VS
    VS --> R[Retrieved Chunks]
    R --> LLM[LLM + Prompt]
    Q --> LLM
    LLM --> A[Answer + Citations]

Implementation Steps

  1. Document Ingestion: Load documents from files (PDF, TXT, HTML)
  2. Chunking: Split into overlapping chunks (512 tokens, 20% overlap)
  3. Embedding: Convert chunks to vectors using an embedding model
  4. Storage: Store vectors in a vector database (or in-memory for prototyping)
  5. Retrieval: Find top-k most similar chunks to the query
  6. Generation: Feed retrieved context + query to an LLM for answer synthesis
Evaluation: Use RAGAS metrics: faithfulness (is the answer supported by context?), answer relevancy (does it address the query?), context precision (are retrieved chunks relevant?).

Step-by-Step Walkthrough

The code above implements each stage of the RAG pipeline. Here's what to focus on at each step:

  1. Chunking strategy matters: The 50-word chunks with 10-word overlap are intentionally small for this demo. In production, use 256-512 tokens with 10-20% overlap. Experiment with recursive character splitting for better semantic boundaries.
  2. Embedding quality: We use TF-based embeddings here (bag-of-words). In production, use sentence-transformers (e.g., all-MiniLM-L6-v2) or OpenAI's text-embedding-3-small for much better semantic matching.
  3. Retrieval tuning: Top-k selection is just the beginning. Consider: re-ranking retrieved chunks with a cross-encoder, filtering by metadata, and using hybrid search (keyword + semantic).
  4. Prompt construction: The generation step should include clear instructions: "Answer ONLY based on the provided context. If the context doesn't contain the answer, say so."

Extensions to Try

  • Add metadata filtering — filter chunks by document type, date, or source before retrieval
  • Implement query expansion — generate multiple rephrased queries to improve recall
  • Add citation tracking — map each sentence in the answer back to its source chunk
  • Build a feedback loop — let users rate answers and use ratings to improve retrieval

Key Takeaways

  • RAG pipeline: ingest → chunk → embed → store → retrieve → generate
  • Overlapping chunks prevent losing context at boundaries
  • Cosine similarity finds the most relevant chunks for a query
  • Evaluate with RAGAS: faithfulness, relevancy, precision, recall

Step-by-Step Walkthrough

  • Use structured handoffs between agents for reliability and debuggability
  • Each agent should have single responsibility with its own error handling
  • An orchestrator coordinates the pipeline and manages agent lifecycle

Step-by-Step Walkthrough

  1. Intent classification: The keyword-based classifier is a simplified version. In production, use an LLM or fine-tuned classifier for nuanced intent detection. Consider handling multi-intent messages (e.g., "check my order and what's the return policy?").

Key Takeaways

  • Knowledge base search handles common questions without LLM calls

Emerging Patterns
