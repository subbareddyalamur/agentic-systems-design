1. LLM Internals

Why this matters: To build production AI systems, you need to understand how LLMs actually work under the hood — tokenization, attention, sampling — so you can debug issues and optimize performance.

The Transformer Architecture

All modern LLMs are based on the Transformer architecture (Vaswani et al., 2017). Key components:

flowchart TD
  I[Input Text] --> T[Tokenizer]
  T --> E[Token Embeddings + Positional Encoding]
  E --> A1[Self-Attention Layer 1]
  A1 --> F1[Feed-Forward Layer 1]
  F1 --> A2[Self-Attention Layer 2]
  A2 --> F2[Feed-Forward Layer 2]
  F2 --> AN[... N layers ...]
  AN --> O[Output Probabilities]
  O --> S[Sampling / Decoding]
  S --> R[Generated Token]
  style I fill:#38bdf8,color:#000
  style R fill:#4ade80,color:#000

Tokenization

LLMs don't see characters or words — they see tokens. A tokenizer splits text into subword units:

  • "Hello world" → ["Hello", " world"] (2 tokens)
  • "unbelievable" → ["un", "believ", "able"] (3 tokens)
  • Common words = 1 token; rare words = multiple tokens

BPE (Byte Pair Encoding) is the most common algorithm. It starts with individual bytes and merges the most frequent pairs iteratively.

Self-Attention

The core mechanism that lets models understand context. Each token "attends" to every other token, learning which words are most relevant to each other:

  • "The cat sat on the mat" — "mat" attends strongly to "sat" and "on"
  • Multi-head attention: multiple attention patterns run in parallel
  • Computational cost: O(n²) where n = sequence length — this is why context windows are limited

Sampling & Temperature

After the model computes probabilities for the next token, a sampling strategy picks the actual token:

  • Temperature: Controls randomness. 0 = always pick highest probability (deterministic). 1 = sample from distribution. >1 = more random.
  • Top-K: Only consider the K most likely tokens. K=1 = greedy, K=50 = moderate diversity.
  • Top-P (nucleus): Consider tokens until cumulative probability reaches P. P=0.9 is common.
Production tip: Use temperature=0 for factual tasks (extraction, classification). Use temperature=0.7-1.0 for creative tasks (writing, brainstorming).

2. Version Control & Rate Limits for LLM Apps

Why this matters: LLM APIs have rate limits, costs, and versioning quirks that don't exist in traditional software. Mismanaging these in production leads to outages, runaway costs, and broken deployments.

API Versioning

LLM providers frequently update models. Production apps must handle this:

  • Pin model versions: Use gpt-4-0613 not gpt-4 (alias changes without notice)
  • Track deprecation dates: Models get sunset — have a migration plan
  • Test before upgrading: New model versions can behave differently
  • A/B test models: Compare old vs. new model on real traffic

Rate Limits

Every LLM API has rate limits (requests per minute, tokens per minute). Key strategies:

  • Exponential backoff: When rate-limited, wait 1s → 2s → 4s → 8s before retrying
  • Request queuing: Queue requests and process at the allowed rate
  • Token budgeting: Track token usage per request to stay within limits
  • Tiered limits: Different API tiers have different limits — plan accordingly

Cost Management

  C -->|Hit| R[Return Cached]
  C -->|Miss| T[Token Estimation]
  T --> B{Within Budget?}
  B -->|No| E[Reject / Queue]
  B -->|Yes| A[Call LLM API]
  A --> S[Store in Cache]
  S --> R

  • Prompt caching: Cache identical or near-identical requests
  • Shorter prompts: Every token costs money — trim unnecessary context
  • Model routing: Use cheaper models (GPT-3.5, Haiku) for simple tasks, expensive models (GPT-4, Opus) for complex ones
  • Usage dashboards: Track spend per feature, per user, per day

3. Retrieval & Grounding

Why this matters: Grounding means ensuring LLM outputs are based on verifiable facts, not hallucinations. Advanced retrieval strategies make the difference between a demo and a production system.

Beyond Basic RAG

Basic RAG (embed → search → generate) works for demos, but production systems need more sophisticated retrieval:

Hybrid Search

Combine vector search (semantic) with keyword search (exact match) for best results:

  • Vector search: "How to fix authentication" finds "resolve login issues"
  • Keyword search: Finds exact terms like error codes, product names
  • Hybrid: Weighted combination of both — catches both semantic and exact matches

Re-ranking

Initial retrieval is fast but rough. A re-ranker (cross-encoder model) scores each retrieved chunk more accurately:

flowchart LR
  Q[Query] --> R1[Initial Retrieval
Top 20 chunks]
  R1 --> RR[Re-ranker Model]
  RR --> R2[Re-ranked
Top 5 chunks]
  R2 --> LLM[Generate Answer]
  style Q fill:#38bdf8,color:#000
  style LLM fill:#4ade80,color:#000

Query Transformation

The user's query may not be optimal for retrieval. Transform it:

  • Query expansion: Add synonyms and related terms
  • HyDE: Generate a hypothetical answer, then search for similar documents
  • Multi-query: Rewrite the query from different angles, search each, combine results
  • Step-back prompting: Ask a more general question first to get broader context

Grounding Techniques

  • Citation extraction: Force the LLM to cite which documents support each claim
  • Faithfulness checking: Verify the answer is supported by retrieved context
  • Confidence scoring: Have the LLM rate its own confidence
  • Source attribution: Link each answer back to its source document

Evaluation Metrics for RAG

  • Context relevance: Are the retrieved chunks relevant to the query?
  • Faithfulness: Is the answer supported by the retrieved context?
  • Answer relevance: Does the answer actually address the question?

4. Advanced RAG Patterns

Why this matters: Basic RAG retrieves chunks and generates answers. Advanced RAG handles complex queries, multi-hop reasoning, tables, and dynamic knowledge — the patterns that separate production systems from prototypes.

Multi-Hop RAG

Some questions require information from multiple documents combined through reasoning:

Q: "What's the GDP per capita of the country where Python was created?"
Step 1: Retrieve → "Python was created in the Netherlands"
Step 2: Retrieve → "Netherlands GDP: $1.01T, Population: 17.5M"
Step 3: Calculate → $1.01T / 17.5M ≈ $57,714

Agentic RAG

Instead of a fixed retrieve→generate pipeline, an agent decides what to retrieve and when:

flowchart TD
  Q[User Query] --> A{Agent: Need retrieval?}
  A -->|Yes| R[Retrieve from knowledge base]
  A -->|No| D[Answer directly]
  R --> E{Enough context?}
  E -->|No| RQ[Reformulate query]
  RQ --> R
  E -->|Yes| G[Generate answer with citations]
  D --> O[Output]
  G --> O
  style Q fill:#38bdf8,color:#000
  style O fill:#4ade80,color:#000

Advanced Patterns

Parent-Child Retrieval

Embed small chunks (for precision) but return the parent chunk (for context). A sentence matches the query, but you return the full paragraph.

Contextual Compression

After retrieval, use a smaller model to extract only the relevant sentences from each chunk, reducing noise in the context.

Self-RAG

The model decides whether to retrieve, generates an answer, then self-evaluates whether the answer is grounded. If not, it retrieves again.

Knowledge Graphs + RAG

Combine structured knowledge (entities, relationships) with unstructured text for more accurate multi-hop reasoning.

Metadata Filtering

Don't just search by content — filter by metadata first:

  • Date range: "Only search documents from last 30 days"
  • Source type: "Only search technical documentation"
  • Access level: "Only return documents the user can access"
  • Language: "Only search English documents"

5. Agent Communication Protocols

Why this matters: Production systems often need multiple agents working together. How agents communicate, share state, and coordinate determines whether the system is reliable or chaotic.

Multi-Agent Systems

Instead of one monolithic agent, production systems often use specialized agents that collaborate:

  • Supervisor pattern: A coordinator agent delegates tasks to specialist agents
  • Peer-to-peer: Agents communicate directly, passing messages
  • Hierarchical: Agents organized in layers (manager → team leads → workers)
flowchart TD
  S[Supervisor Agent] --> R[Research Agent]
  S --> W[Writer Agent]
  S --> C[Code Agent]
  R -->|findings| S
  W -->|draft| S
  C -->|code| S
  S --> U[Final Output to User]
  style S fill:#38bdf8,color:#000
  style U fill:#4ade80,color:#000

Communication Patterns

1. Message Passing

Agents send structured messages to each other. Each message has a sender, receiver, content, and type:

{
  "from": "research_agent",
  "to": "writer_agent",
  "type": "research_complete",
  "content": {"findings": [...], "sources": [...]}
}

2. Shared State

All agents read from and write to a shared state object (like LangGraph's state). Simpler but requires careful state management to avoid conflicts.

3. Event-Driven

Agents publish events and subscribe to event types they care about. Loosely coupled — agents don't need to know about each other.

Coordination Challenges

  • Race conditions: Two agents modifying state simultaneously
  • Deadlocks: Agents waiting on each other
  • Infinite loops: Agents triggering each other endlessly
  • Error propagation: One agent's failure cascading to others
Best practice: Use explicit turn-taking or a supervisor to prevent race conditions. Set maximum iteration limits to prevent infinite loops.

6. Output Validation & Guardrails

Why this matters: LLMs can produce invalid JSON, harmful content, hallucinated data, or off-topic responses. Guardrails catch these issues before they reach users or downstream systems.

Types of Validation

1. Structural Validation

Ensure the output matches the expected format:

  • Valid JSON/YAML
  • Matches Pydantic schema
  • Required fields present
  • Correct data types

2. Content Validation

  • No harmful/toxic content
  • No PII (personally identifiable information) leakage
  • Stays on topic (relevance check)
  • Factual consistency with source documents

3. Business Logic Validation

  • Prices are within expected range
  • Dates are in the future (for bookings)
  • Recommendations match user preferences

Guardrail Architecture

flowchart LR
  I[User Input] --> IG[Input Guard]
  IG --> LLM[LLM Processing]
  LLM --> OG[Output Guard]
  OG -->|Pass| U[User]
  OG -->|Fail| R[Retry / Fallback]
  R --> LLM
  style I fill:#38bdf8,color:#000
  style U fill:#4ade80,color:#000
  style R fill:#fbbf24,color:#000

Implementation Strategies

  • Regex patterns: Check for PII (emails, phone numbers, SSNs)
  • Schema validation: Pydantic, JSON Schema for structural checks
  • Classifier models: Small models that classify toxicity, relevance
  • LLM-as-judge: Use a second LLM to evaluate the first LLM's output
  • Rule-based: Hardcoded business rules (price ranges, allowed categories)

Retry Strategies

  • Simple retry: Call the LLM again with the same prompt
  • Retry with feedback: Include the error message in the next prompt
  • Fallback model: Try a different model if the first fails
  • Graceful degradation: Return a safe default response

7. Memory Architecture for Production Agents

Why this matters: Production agents need sophisticated memory — tracking users, conversations, and learned facts across sessions, while staying within context limits and keeping responses fast.

Production Memory Requirements

  • Persistence: Survive server restarts and deployments
  • Scalability: Handle thousands of concurrent users
  • Latency: Memory retrieval must be fast (<100ms)
  • Privacy: Each user only sees their own data
  • Consistency: Memory updates must be atomic and reliable

Memory Architecture

flowchart TD
  A[Agent] --> CM[Context Manager]
  CM --> STM[Short-Term Memory
Redis / In-memory]
  CM --> LTM[Long-Term Memory
Vector DB]
  CM --> EM[Entity Memory
PostgreSQL]
  CM --> PM[Procedural Memory
Learned patterns]
  STM --> W[Context Window
Recent messages]
  LTM --> W
  EM --> W
  style A fill:#38bdf8,color:#000
  style W fill:#4ade80,color:#000

Memory Tiers

Tier 1: Working Memory (in-context)

The current conversation. Fits in the LLM's context window. Fast but limited.

Tier 2: Session Memory (Redis/cache)

Full conversation history for the current session. Retrieved on demand. Millisecond latency.

Tier 3: User Memory (database)

Per-user facts, preferences, and history. Persists across sessions. Queried by user ID.

Tier 4: Knowledge Memory (vector store)

Organization-wide knowledge base. Shared across users. Retrieved by semantic similarity.

Context Window Management

The context window is limited. Strategies to manage it:

  • Sliding window: Keep last N messages, drop oldest
  • Summarization: Periodically summarize old messages
  • Selective retrieval: Only bring in relevant memories
  • Importance scoring: Keep important messages, drop routine ones

8. Advanced Orchestration

Why this matters: Real-world AI systems need to orchestrate multiple LLM calls, tool executions, and human approvals in complex workflows. Advanced orchestration patterns make these systems reliable and maintainable.

Orchestration Patterns

1. Chain of Thought Orchestration

Break complex reasoning into explicit steps, each handled by a specialized prompt or model:

Step 1: Extract entities (cheap model)
Step 2: Classify intent (cheap model)
Step 3: Generate response (expensive model, only if needed)
Step 4: Validate output (guardrail)

2. Map-Reduce

Split a large task into parallel subtasks, process each, then combine results:

  • Map: Summarize each of 10 documents in parallel
  • Reduce: Combine all summaries into one final summary

3. Router Pattern

A classifier routes requests to the right handler — different models, prompts, or tools for different request types.

4. Human-in-the-Loop

Pause the workflow at critical points for human approval before continuing:

flowchart LR
  A[Agent Draft] --> R{Risk Level?}
  R -->|Low| E[Execute Directly]
  R -->|Medium| N[Notify Human]
  R -->|High| W[Wait for Approval]
  N --> E
  W -->|Approved| E
  W -->|Rejected| M[Modify & Retry]
  M --> A
  style A fill:#38bdf8,color:#000
  style E fill:#4ade80,color:#000

Workflow State Management

  • Checkpointing: Save workflow state at each step so you can resume on failure
  • Idempotency: Make operations safe to retry without side effects
  • Timeouts: Set per-step timeouts to prevent hanging workflows
  • Cancellation: Allow users to cancel long-running workflows

9. Resilience & Fallbacks

Why this matters: LLM APIs go down. Models hallucinate. Network timeouts happen. Production systems must handle failures gracefully — users should never see a raw error.

Failure Modes in LLM Systems

  • API outages: Provider is down (OpenAI, Anthropic, etc.)
  • Rate limiting: Too many requests
  • Timeout: Model takes too long to respond
  • Invalid output: Model returns malformed JSON or nonsense
  • Hallucination: Model generates factually incorrect content
  • Context overflow: Input exceeds model's context window

Resilience Patterns

Circuit Breaker

After N consecutive failures, stop calling the failing service. Periodically test if it's recovered:

  • Closed: Normal operation — requests pass through
  • Open: Too many failures — redirect to fallback
  • Half-open: Let one test request through to check recovery

Graceful Degradation

  • Return cached results when the API is down
  • Use a simpler model when the primary is unavailable
  • Show partial results rather than nothing
  • Queue requests for later processing

Timeout Management

  • Per-request timeouts: Don't wait forever for a response (30-60s typical)
  • Streaming: Use streaming responses so users see partial output immediately
  • Background processing: For long tasks, process async and notify when done

10. Tracing & Debugging Agents

Why this matters: When an agent gives a wrong answer or gets stuck in a loop, you need to trace exactly what happened — every LLM call, tool use, and decision point. Without tracing, debugging AI systems is nearly impossible.

Why Tracing Is Essential

Traditional debugging (print statements, breakpoints) doesn't work well for LLM agents because:

  • LLM calls are non-deterministic — same input can give different outputs
  • Agent loops may run for many iterations before the bug appears
  • Tool calls have side effects that compound
  • Context window changes at each step affect all subsequent steps

Observability Stack

flowchart TD
  A[Agent Execution] --> T[Traces]
  A --> L[Logs]
  A --> M[Metrics]
  T --> D[LangSmith / LangFuse / Phoenix]
  L --> E[ELK / CloudWatch]
  M --> G[Grafana / Datadog]
  D --> DB[Debug Dashboard]
  E --> DB
  G --> DB
  style A fill:#38bdf8,color:#000
  style DB fill:#4ade80,color:#000

What to Trace

  • LLM calls: Input prompt, output, model, temperature, tokens used, latency
  • Tool calls: Tool name, arguments, result, latency
  • Retrieval: Query, retrieved documents, relevance scores
  • Decisions: Why the agent chose a particular path
  • State changes: Before and after snapshots of agent state

Tracing Tools

  • LangSmith: Official LangChain tracing platform. Rich UI for inspecting chains.
  • LangFuse: Open-source alternative. Self-hostable.
  • Phoenix (Arize): Open-source observability for LLM apps.
  • OpenTelemetry: Industry-standard tracing, adapted for LLM apps.

Debugging Strategies

  • Replay traces: Re-run a specific trace with the exact same inputs
  • Compare traces: Diff a successful vs. failed trace to find the divergence
  • Step-through: Execute the agent step-by-step, inspecting state at each point
  • Prompt debugging: Extract the exact prompt that caused the issue and test in isolation

11. Evaluation & Testing AI Systems

Why this matters: You can't improve what you can't measure. Evaluating LLM systems requires different approaches than traditional software testing — you need metrics for quality, safety, and reliability.

Types of Evaluation

1. Unit-Level Evaluation

  • Test individual LLM calls with known input/output pairs
  • Validate tool calls return expected results
  • Check prompt templates produce valid output

2. Component-Level Evaluation

  • RAG retrieval quality (precision@k, recall@k)
  • Agent decision accuracy (does it choose the right tool?)
  • Memory recall accuracy (does it remember the right facts?)

3. System-Level Evaluation

  • End-to-end task completion rate
  • User satisfaction scores
  • Latency and cost per request

12. SLIs, SLOs & Monitoring

Why this matters: Production AI systems need measurable reliability targets. SLIs (what to measure) and SLOs (target values) let you know when your system is healthy — and when to sound the alarm.

Key Definitions

  • SLI (Service Level Indicator): A measurable metric — e.g., "response latency in ms"
  • SLO (Service Level Objective): A target for an SLI — e.g., "p95 latency < 3s"
  • SLA (Service Level Agreement): A contract with consequences — e.g., "99.9% uptime or credits issued"
  • Error Budget: How much failure your SLO allows — e.g., 99.9% uptime = 43 minutes of downtime/month

14. Ops: Caching & Concurrency

Why this matters: LLM API calls are slow (1-30s) and expensive. Caching avoids redundant calls. Concurrency handles multiple users simultaneously. Together, they make your AI system fast and cost-effective.

Caching Strategies

Exact Match Cache

Hash the prompt and check if you've seen it before. Fast and simple but only catches identical requests.

Semantic Cache

Embed the prompt and search for similar cached prompts. Catches paraphrases ("What's ML?" ≈ "Explain machine learning").

Cache Layers

  • L1 (in-memory): Dict/LRU cache in the process. Fastest, lost on restart.
  • L2 (Redis): Shared cache across instances. Persists across restarts. Millisecond latency.
  • L3 (database): Permanent cache. Survives everything. Slower but unlimited storage.

Cache Invalidation

  • TTL (Time-To-Live): Expire after N hours/days
  • Version-based: Invalidate when the model version changes
  • Event-based: Invalidate when source documents are updated

Concurrency

Async/Await

Python's asyncio lets you make multiple LLM calls concurrently without blocking:

import asyncio

async def call_llm(prompt):
    response = await client.chat.completions.create(...)
    return response

# Run 5 calls concurrently instead of sequentially
results = await asyncio.gather(
    call_llm("prompt1"),
    call_llm("prompt2"),
    call_llm("prompt3"),
    call_llm("prompt4"),
    call_llm("prompt5"),
)

Connection Pooling

Reuse HTTP connections to the LLM API instead of creating new ones for each request. Reduces latency and resource usage.

Rate-Limit-Aware Concurrency

Use semaphores to limit concurrent calls to match your API rate limit:

semaphore = asyncio.Semaphore(10)  # Max 10 concurrent calls

async def rate_limited_call(prompt):
    async with semaphore:
        return await call_llm(prompt)

15. Maintenance & Disaster Recovery

Why this matters: AI systems degrade over time — models drift, APIs change, and knowledge bases go stale. A maintenance plan and disaster recovery strategy keep your system reliable for the long term.

Model Drift

Over time, LLM behavior changes even with the same prompt. Causes:

  • Provider updates: Model weights change with fine-tuning updates
  • World changes: The real world changes but your prompts/data don't
  • User drift: User behavior changes over time
  • Data staleness: Your knowledge base becomes outdated

TaskFrequencyWhy

Disaster Recovery

Common Disasters

  • API provider outage: OpenAI/Anthropic down for hours
  • Model deprecation: Your pinned model version is sunset
  • Data loss: Vector store corrupted or deleted
  • Prompt injection attack: Malicious users exploit your system
  • Cost explosion: Bug causes runaway API spending

DR Playbook

  • Multi-provider setup: Have fallback providers (OpenAI + Anthropic + local model)
  • Regular backups: Back up vector store, configurations, and evaluation datasets
  • Spending alerts: Set hard caps and alerts on API spend
  • Kill switch: Ability to instantly disable the AI system
  • Rollback plan: Quickly revert to a previous working configuration
